# Plan 7.2: Backend Enforcement (Revised)

## Objective
Enforce the granular Analytics Access Control server-side. 
1. `Summary/Graphs` (Dashboard API): Allow global access if `is_analytics_enabled`.
2. `Responses` API: Allow global access if `is_analytics_enabled`, but strip classifications for non-superadmins when in global view.
3. `Data View` (specific offices): Strictly restrict to assigned offices even if `is_analytics_enabled`.

## Context
- .gsd/FINALrev.md
- src/lib/auth/rbac.ts
- src/app/api/responses/route.ts
- src/app/api/dashboard/route.ts

## Tasks

<task type="auto">
  <name>Revise RBAC Scoping Logic</name>
  <files>src/lib/auth/rbac.ts</files>
  <action>
    Update `resolveAuthorizedOffices(user, requestedOffices)`:
    1. If `hasGlobalAccess(user)` (Superadmin), allow anything.
    2. If `user.is_analytics_enabled === true`:
       - If `requestedOffices` is `["ALL"]` or empty/missing (defaulting to ALL), allow `["ALL"]`.
       - If `requestedOffices` contains specific office IDs, **strictly intersect** with `user.offices`. 
       - This ensures that while they can see "ALL", they cannot pick-and-choose specific offices they don't own (preventing horizontal escalation in granular views).
    3. If `!is_analytics_enabled`, strictly map "ALL" to `user.offices` and intersect everything else.
  </action>
  <verify>Check rbac.ts for intersection logic even when analytics is enabled</verify>
  <done>RBAC correctly differentiates between global request and granular office request for analytics users</done>
</task>
