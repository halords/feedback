## Tasks

<task type="auto">
  <name>Implement Security Headers</name>
  <files>src/next.config.ts (or standard next.config.js)</files>
  <action>
    - Add the `headers()` async configuration to Next.js config.
    - Implement `Content-Security-Policy` (CSP). Since the app uses inline scripts for hydration/styles, use a relaxed but protective baseline (e.g., `default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https://firebasestorage.googleapis.com;`).
    - Implement `X-Frame-Options: DENY`.
    - Implement `X-Content-Type-Options: nosniff`.
    - Implement `Referrer-Policy: strict-origin-when-cross-origin`.
    - **Avoid**: Extremely strict CSP that breaks Firebase Authentication or React hydration.
  </action>
  <verify>curl -I http://localhost:3000</verify>
  <done>HTTP responses include the specified security headers.</done>
</task>

<task type="auto">
  <name>Rate Limit Heavy Endpoints</name>
  <files>src/app/api/reports/bulk/route.ts, src/app/api/reports/summary/route.ts</files>
  <action>
    - Import `checkRateLimitAsync` from `src/lib/security/rateLimit.ts`.
    - Add rate limiting logic at the beginning of the `POST` handlers in both routes.
    - Set a restrictive limit (e.g., max 5 requests per 5 minutes or similar logic depending on the current rate limiting schema setup, perhaps 10 requests per 60000ms window) because PDF generation consumes significant memory.
    - Return `429 Too Many Requests` when limits are exceeded.
  </action>
  <verify>curl -X POST http://localhost:3000/api/reports/bulk (Observe 429 after rapid loops if possible, or verify code injection is valid)</verify>
  <done>Heavy PDF generation APIs successfully throttle requests using the stateful Firestore limiter.</done>
</task>
